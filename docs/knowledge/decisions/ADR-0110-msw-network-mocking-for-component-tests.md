# ADR-0110: MSW network mocking for component tests

## Status

Accepted, landed. Component tests can now mock the network. The change adds `msw` and
`msw-storybook-addon`, starts the Mock Service Worker from the Storybook preview, and
ships a small module of shared request handlers that both the browser-mode stories and
the node unit suite reuse. The furniture library renders in a story against a mocked
pack manifest, in its success, empty, and error states, with no real network read. This
record extends ADR-0105, which built the browser-mode harness that this work plugs into.

## Context

ADR-0105 turned stories into browser-mode component tests but left network mocking as
named follow-on work. Data-driven components could not render deterministically in
isolation, because the only way to populate them was to let a real `fetch` reach a real
file. The furniture library is the clear example: it lists assets read from a pack
served as static files, so without a mock it can only show whatever is on disk, and its
empty and error states have no way to appear in a test at all.

A second question had to be answered before wiring anything: what does the application
actually fetch? The issue framed the surface broadly, naming the pack reader, underlay
image loaders, and recent projects. Reading the code narrows it sharply. Only the pack
reader uses `fetch`. The underlay loaders read a user-picked `File` and decode it with
`createImageBitmap`, recent projects come from IndexedDB, the SVG export rasterizes
through a `blob:` URL on an `Image`, and file import uses a `FileReader`. None of those
touch the network, so none of them are interceptable by a service worker. The mockable
surface is the pack manifest and its asset and thumbnail bytes under `/packs`, and
nothing else.

## Decision

Add MSW, start it from the preview, and keep the request handlers in one shared module
that serves a valid pack manifest. Scope the mocking to the pack `fetch` surface, since
that is the only network surface there is.

### The two dependencies, exact-pinned and past cooldown

`msw` is pinned at 2.14.6 and `msw-storybook-addon` at 2.0.7. Both clear the 30-day
install cooldown: `msw` 2.14.6 was published on 2026-05-11 and the addon 2.0.7 on
2026-04-08, with today at 2026-06-19. The addon peers `msw` at `^2.0.0`, satisfied by
2.14.6. The install was run surgically with
`--config.prefer-frozen-lockfile=true --config.minimumReleaseAge=0`, the pattern
ADR-0100 and ADR-0105 established, and the lockfile diff was checked to confirm only the
two intended keys and their pinned transitives moved.

### The worker starts from the preview, and the loader was already applied to the browser project

`.storybook/preview.ts` calls `initialize` once at module load and adds `mswLoader` to
the preview loaders. Per-story handlers live in `parameters.msw.handlers`. The worker
script `public/mockServiceWorker.js` is generated by `msw init public/ --save` and
committed, so a fresh clone, Storybook dev, and the browser-mode test run all register
the same interceptor. The worker version tracks the installed `msw` version and is
regenerated on any bump.

ADR-0105 noted that since Storybook 10.3 the Vitest addon applies the preview
annotations to the browser project on its own. The open question for this change was
whether loaders count among those annotations, since a loader is what starts MSW per
story. A spike answered it before any commit: a story with mocked handlers ran in
chromium under the browser project, the worker intercepted the manifest request, and the
panel rendered the mocked assets. So no `setProjectAnnotations` setup file is needed and
`vitest.config.ts` stays byte-identical, which keeps the Stryker config resolution that
ADR-0100 documented untouched.

`initialize` is called with `onUnhandledRequest: 'bypass'` and `quiet: true`. Bypass lets
the Storybook runtime's own asset requests pass through, so the suite does not fail on
requests that are not the point of a given story. Quiet keeps the matched-request logging
out of the gate output. Bypassing unhandled requests gives up the strongest form of
"no real network" proof, so each networked story earns that proof a different way: it
asserts a result only the mock can produce. The on-disk manifest holds a single asset
named "Example chair", so a success story that renders two differently named buttons can
only have been served by the mock.

### One shared handlers module, used from both runtimes

`storage/assets/msw-pack-handlers.ts` exports `packHandlers`, which serves a valid pack
manifest listing exactly the assets it is given, and `packErrorHandlers`, which serves a
non-ok manifest response. The module lives in `storage/` next to the fetch pack reader so
both the editor stories and the storage unit test import it from one source of truth. It
is code under test, not a throwaway fixture: it encodes the application's real pack URL
contract, and it imports only `msw` and pure data, so it respects the layer rules.

The two runtimes reach the same handlers through different bases. The browser story uses
the relative base `/packs/vernacular-starter-1.0.0`, which the worker resolves against the
Storybook iframe origin. The node unit test uses an absolute base, because node's `fetch`
rejects a relative URL and `msw/node` matches absolute request URLs; it installs the
handlers on a `setupServer` and drives the real `PackSource` to confirm the manifest it
serves lists as expected. The empty and error states are both exercised: an empty but
valid manifest lists nothing, and a non-ok manifest reads as an empty pack rather than a
thrown render.

## Consequences

- Networked components can now be tested in isolation. The furniture library renders its
  success, empty, and error states from mocked pack responses with no real network, which
  is the concrete deliverable, and the shared handlers are ready for the story backfill and
  visual-regression work that follow.
- The mocking layer is deliberately scoped to `fetch`. Underlay images, recent projects,
  SVG rasterization, and file import are local reads through `File`, IndexedDB, `blob:`
  URLs, and `FileReader`, so a service worker cannot intercept them. Stories for those
  components will need their own seams rather than MSW, and this record names that boundary
  so a later contributor does not look for a network mock that cannot exist.
- The worker is a committed artifact tied to the `msw` version. A missing or stale
  `public/mockServiceWorker.js` silently disables interception and a story would read the
  real disk, so it is regenerated whenever `msw` is bumped.
- The browser project still applies the preview annotations on its own. Loaders are among
  them, confirmed in practice, so this change adds no setup file and leaves
  `vitest.config.ts` and the Stryker config path alone.
